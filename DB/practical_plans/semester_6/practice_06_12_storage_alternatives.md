# Практическое занятие 6.12. Реляционное хранение и альтернативы внутри PostgreSQL

Место в курсе: занятие готовит architecture trade-off review. Студент сравнивает нормализованную схему, JSONB, key-value pattern и object reference для конкретного класса данных.

Тип занятия: практика.

## Цели

- Выбрать класс данных, для которого есть соблазн уйти из реляционной модели.
- Сравнить нормализованную реляционную модель, PostgreSQL JSONB, key-value pattern и object storage reference по требованиям.
- Определить, что должно остаться транзакционным и авторитетным.
- Подготовить аргументы для ADR.

## Входные условия

- Студент понимает различие между источником достоверных данных, производным индексом, кешем и внешним хранилищем.
- Есть один сценарий: пользовательские настройки, диагностический payload, event metadata, document-like entity или крупный artifact reference.
- Преподаватель задает требования: consistency, query patterns, size, retention, source of truth.

## План занятия

1. Выбор класса данных.
2. Описание access patterns: write, read, search, retention, export.
3. Определение consistency requirements.
4. Сравнение normalized relational, PostgreSQL JSONB, key-value pattern и object storage reference.
5. Определение authoritative source.
6. Обсуждение operational burden.
7. Заполнение trade-off table.
8. Подготовка ADR-черновика для недели 13.

## Задание

Студент должен подготовить сравнительный анализ модели хранения.

Минимальный состав:

- описание данных и владельца;
- 3-5 access patterns;
- таблица сравнения минимум трех вариантов;
- выбранный вариант;
- явно указанные non-goals;
- последствия для миграций, backup, поиска и эксплуатации.

## Критерии зачета

- Сравнение опирается на требования, а не на моду технологии.
- Авторитетный источник данных определен явно.
- Выбранный вариант имеет описанные ограничения.
- ADR показывает, какие альтернативы были отвергнуты и почему.

## Контрольные вопросы: что вы должны уметь после занятия

1. Когда PostgreSQL с JSONB лучше отдельной document DB?
2. Когда object storage лучше таблицы БД?
3. Какие данные должны оставаться транзакционными?
4. Как access pattern влияет на выбор хранилища?
5. Что значит authoritative source?
6. Какие эксплуатационные вопросы нужно задать перед выбором нового хранилища?
