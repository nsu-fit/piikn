# Практическое занятие 5.14. Безопасный SQL из приложения

Место в курсе: занятие следует после лекции по безопасному SQL и готовит интегрированную контрольную 3.

Тип занятия: практика.

## Цели

- Исправлять небезопасную строковую сборку SQL.
- Разделять параметризуемые значения и динамические идентификаторы.
- Объяснять, какие права нужны приложению для выполнения SQL-сценария.
- Проверять SQL-фрагмент на типовые injection-паттерны.

## Входные условия

- Выполнена практика по транзакциям.
- Прослушана лекция по безопасному SQL из приложения.
- Преподаватель выдает фрагменты псевдокода или SQL-шаблонов.

## План занятия

1. Разбор небезопасного SQL-фрагмента.
2. Параметризация значений.
3. Prepared statement и placeholders на уровне псевдокода.
4. Проверка динамического SQL: whitelist для имен таблиц, столбцов и направлений сортировки.
5. Минимальные права приложения: что нужно для чтения, изменения и служебных операций.
6. Проверка исправления на типовых payload.
7. Разбор границы: SQL injection, authorization bug и ошибка бизнес-логики.
8. Подготовка к контрольной 3.

## Задание

Студент должен исправить 2-3 небезопасных SQL-шаблона и объяснить, какие атаки или ошибки становятся невозможными.

Обязательные элементы:

- безопасный вариант запроса с параметрами;
- объяснение, что нельзя параметризовать обычным placeholder-ом;
- краткое описание минимальных прав приложения.
- проверка исправления на 1-2 типовых payload.

## Артефакт сдачи

Папка `semester-5/practice-14-sql-security`:

- `secure_sql.md`;
- при необходимости `fixed_queries.sql`.

## Критерии зачета

- Исправленный SQL не собирает пользовательские значения строковой конкатенацией.
- Динамические идентификаторы проверяются через whitelist.
- Студент понимает, какие права нужны приложению и почему не стоит работать от имени владельца схемы.
- Объяснение различает injection, отсутствие авторизации и ошибку бизнес-правила.

## Контрольные вопросы: что вы должны уметь после занятия

1. Почему строковая сборка SQL с пользовательским вводом опасна?
2. Почему параметризация защищает значения, но не имена столбцов?
3. Как безопасно реализовать динамическую сортировку?
4. Какие права обычно нужны приложению для выполнения SQL-сценария?
5. Чем SQL injection отличается от ошибки авторизации?
6. Какие признаки небезопасного SQL нужно искать на code review?
