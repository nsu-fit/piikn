# Практическое занятие 5.9. DML-сценарии на основе SELECT

Место в курсе: занятие следует после лекции по DML и контрольной по базовому `SELECT`. Оно показывает, как команды изменения данных используют запросы для выбора источника, ограничения изменяемых строк и проверки результата.

Тип занятия: практика.

## Цели

- Выполнять `INSERT`, `UPDATE`, `DELETE` в рамках заданного сценария.
- Использовать `INSERT ... SELECT`, `UPDATE ... FROM`, `DELETE ... USING` и `RETURNING`.
- Проверять инварианты до и после изменения данных.
- Понимать, как CTE помогает сделать сложный DML-сценарий читаемым.

## Входные условия

- Базовый `SELECT` уже отработан на контрольной 2.
- Студент знает DDL и базовые ограничения.
- Работа проводится в изолированной учебной среде, где ошибки не влияют на общую базу.

## План занятия

1. Разбор сценария изменения данных.
2. Предварительный `SELECT`: какие строки и почему будут затронуты.
3. `INSERT ... SELECT` для переноса или материализации результата.
4. `UPDATE ... FROM` с точным условием соединения.
5. `DELETE ... USING` или мягкое отключение записи.
6. `RETURNING` и проверка количества затронутых строк.
7. CTE как способ сделать DML-сценарий читаемым.
8. Проверочные `SELECT` до и после изменения.

## Задание

Студент должен выполнить несколько DML-операций и объяснить результат проверочных запросов.

- предварительный `SELECT`, показывающий целевые строки;
- один `INSERT ... SELECT` или эквивалентный перенос данных;
- один `UPDATE ... FROM`;
- один `DELETE ... USING` или мягкое отключение записи;
- использование `RETURNING`;
- проверка, что изменение не нарушило инварианты.

## Критерии зачета

- DML-операции корректны и не ломают целостность.
- Предварительные `SELECT` действительно соответствуют изменяемым строкам.
- `RETURNING` и проверочные запросы интерпретированы корректно.
- CTE используются для читаемости, а не для сокрытия неясной логики.

2. Как `INSERT ... SELECT` отличается от обычного `INSERT ... VALUES`?

4. Как `RETURNING` помогает проверить результат изменения?
5. Почему DML-сценарий должен быть идемпотентным или явно неидемпотентным?
