# Практические упражнения по дисциплине "Базы данных"

Этот каталог содержит выдаваемые студентам задания, варианты и спецификации проверки. Методические планы занятий находятся в `DB/practical_plans/`; здесь фиксируются конкретные упражнения.

## Базовый формат упражнения

```text
practice_xx_yy/
  statement.md        # выдаваемое задание
  variant.yaml        # параметры вариантов
  provided/           # выданные SQL-сценарии и исходные материалы, если есть
  expected/README.md  # ожидаемые свойства решения
  tests/README.md     # контракт автопроверки
```

## Общие правила

- Основная база 5-го семестра: Postgres Pro demo database "Авиаперевозки", версия `2025-09-01`, схема `bookings`.
- Видимая база для обычной практики: `3 months`.
- Скрытые проверки могут выполняться на `6 months` или `1 year` той же схемы.
- Запрещено подгонять решения под конкретные строки видимой базы: hard-coded `ticket_no`, `book_ref`, `flight_id`, суммы и количества допустимы только если они явно выданы в варианте.
- Если порядок строк важен, в задании должен быть явно указан `ORDER BY`.
- SQL-решения должны быть переносимыми между размерами демобазы.
- В практиках 5.1-5.3 студенты запускают и читают выданные SQL-сценарии, но не пишут SQL самостоятельно.
- Практика 5.5 также не требует самостоятельного SQL: студенты работают с выражениями реляционной алгебры и табличными результатами.
- Самостоятельное написание DDL, прикладных `SELECT` и DML появляется только после соответствующих лекций. Если до DML-лекции нужно проверить ограничения через `INSERT`, такой сценарий выдается преподавателем готовым.

## Структура сдачи студентом

Студент сдает работу в своем репозитории, в папке вида:

```text
semester-5/practice-05-01/
  README.md
  *.sql
  answers.md
```

Конкретные имена файлов задаются в `statement.md` каждого упражнения.

## Проверка

Автоматически проверяются:

- наличие обязательных файлов;
- выполнение SQL без ошибок;
- форма результата: имена столбцов, количество столбцов, базовые инварианты;
- отсутствие очевидного hard-code там, где запрос должен быть обобщенным;
- переносимость на скрытый размер базы, если это указано в тестовом контракте.

Вручную проверяются:

- интерпретация предметной области;
- корректность ER-гипотез;
- объяснение спорных мест схемы;
- качество Markdown-ответов;
- ответы на короткой защите.
