# Практика 5.14. Безопасный SQL из приложения

Вы исправляете небезопасные SQL-шаблоны и объясняете, какую ошибку устраняет каждое исправление. Основной фокус: параметризация значений, whitelist для динамических идентификаторов и минимальные права приложения.

## Что нужно сдать

```text
semester-5/practice-14-sql-security/
  README.md
  secure_sql.md
  fixed_queries.sql
```

## Исходные материалы

Используйте фрагменты из `provided/insecure_fragments.md`.

## Задания

### 1. Поиск заказов по статусу

Исходный фрагмент собирает условие `WHERE status = '...'` через строковую конкатенацию.

В `secure_sql.md`:

- покажите безопасный псевдокод с placeholder-ом;
- объясните, почему payload из варианта больше не меняет структуру SQL;
- укажите, какие значения статуса должны проходить прикладную валидацию.

В `fixed_queries.sql` добавьте безопасный SQL-шаблон с параметром `$1`.

### 2. Динамическая сортировка

Исходный фрагмент подставляет имя столбца и направление сортировки из пользовательского ввода.

В `secure_sql.md`:

- объясните, почему имя столбца нельзя защитить обычным placeholder-ом;
- предложите whitelist для `sort_column` и `sort_direction`;
- покажите итоговый SQL после выбора из whitelist.

### 3. Tenant boundary

Исходный фрагмент ищет документ только по `document_id`.

Исправьте его так, чтобы запрос учитывал `tenant_id` текущего пользователя. Объясните, почему это ошибка авторизации, а не только SQL injection.

### 4. Минимальные права

Опишите три роли:

- owner/migration role;
- application role;
- read-only support role.

Для каждой роли укажите, какие операции ей нужны и какие не нужны.

## Payload для самопроверки

Используйте значения из `variant.yaml`:

- `status_payload`;
- `sort_column_payload`;
- `tenant_document_id`.
