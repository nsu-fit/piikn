# Практика 5.2. Анализ учебного датасета и словарь данных

## Контекст

Вы продолжаете работу с базой Postgres Pro "Авиаперевозки". Теперь нужно прочитать фрагмент схемы как модель предметной области.

На этой неделе вы **не пишете SQL самостоятельно**. Вы запускаете выданный файл `provided/schema_profile.sql`, выбираете часть результата, относящуюся к вашему варианту, и интерпретируете ее.

Каждый студент получает один вариант из `variant.yaml`. Вариант задает группу таблиц для анализа.

## Что нужно сдать

```text
semester-5/practice-05-02/
  README.md
  schema_profile.sql
  data_dictionary.md
  query_observations.md
  answers.md
```

Файл `schema_profile.sql` - копия выданного файла `provided/schema_profile.sql` без изменений.

## Задание

### 1. Запуск готового профиля схемы

1. Запустите `provided/schema_profile.sql`.
2. В `query_observations.md` выпишите наблюдения по таблицам вашего варианта.
3. Не переписывайте весь вывод. Нужны только факты, которые помогают понять предметную область.

### 2. Словарь данных

В `data_dictionary.md` для таблиц вашего варианта опишите:

- назначение таблицы;
- ключевые атрибуты;
- первичные ключи;
- внешние ключи;
- 2-3 бизнес-правила, которые видны из схемы;
- 1-2 неявных или спорных правила, которые не полностью проверяются ограничениями.

Не ограничивайтесь переводом имен столбцов. Нужно объяснить смысл таблиц в предметной области.

### 3. Интерпретация готовых запросов

В `query_observations.md` обязательно должны быть разделы:

- `columns` - что видно по столбцам и типам;
- `constraints` - какие ограничения важны;
- `cardinality` - какие связи и распределения заметны;
- `quality checks` - какие правила проверяют готовые диагностические запросы.

Проверка может вернуть ноль строк. Это нормальный результат, если вы объяснили, какое правило проверялось.

## answers.md

Ответьте:

1. Какой фрагмент предметной области описывают таблицы вашего варианта?
2. Какие связи являются обязательными, а какие допускают отсутствие связанных строк?
3. Какие бизнес-правила выражены ограничениями БД?
4. Какие правила не проверяются схемой напрямую?
5. Какой готовый диагностический запрос кажется самым важным и почему?

## Ограничения

- Не нужно писать свои SQL-запросы.
- Не утверждайте, что правило гарантируется БД, если вы видите его только по текущим данным.
- Если вы используете человекочитаемые названия стран, городов или аэропортов, учитывайте, что в сценарии зафиксирован `SET bookings.lang = 'en'`.
