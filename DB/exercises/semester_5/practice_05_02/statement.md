# Практика 5.2. Анализ учебного датасета и словарь данных

## Контекст

Вы продолжаете работу с базой Postgres Pro "Авиаперевозки". Теперь нужно не просто выполнить запросы, а прочитать фрагмент схемы как модель предметной области.

Каждый студент получает один вариант из `variant.yaml`. Вариант задает:

- группу таблиц;
- основной вопрос анализа;
- дополнительные параметры для SQL-проверок.

Если в варианте используются человекочитаемые названия стран, городов или аэропортов, в начале соответствующего SQL-файла зафиксируйте язык:

```sql
SET bookings.lang = 'en';
```

## Задание

### 1. Словарь данных

В `data_dictionary.md` для таблиц вашего варианта опишите:

- назначение таблицы;
- ключевые атрибуты;
- первичные ключи;
- внешние ключи;
- 2-3 бизнес-правила, которые видны из схемы;
- 1-2 неявных или спорных правила, которые не полностью проверяются ограничениями.

Не ограничивайтесь переводом имен столбцов. Нужно объяснить смысл таблиц в предметной области.

### 2. Столбцы и типы

В `q01_columns.sql` напишите запрос к `information_schema.columns` или `pg_catalog`, который выводит столбцы таблиц вашего варианта.

Обязательные столбцы результата:

- `table_name`;
- `column_name`;
- `data_type`;
- `is_nullable`;
- `ordinal_position`.

Отсортируйте по `table_name`, затем по `ordinal_position`.

### 3. Ограничения

В `q02_constraints.sql` напишите запрос к системным каталогам, который выводит ограничения таблиц вашего варианта.

Обязательные столбцы результата:

- `table_name`;
- `constraint_name`;
- `constraint_type`;
- `constraint_definition`.

Нужно показать первичные ключи, внешние ключи, уникальные ограничения и check-ограничения, если они есть.

### 4. Кардинальности

В `q03_cardinality.sql` напишите 3 запроса, которые помогают понять связи между таблицами вашего варианта.

Примеры допустимых проверок:

- сколько билетов входит в одно бронирование;
- сколько сегментов входит в один билет;
- сколько рейсов относится к одному маршруту;
- сколько мест разных классов есть у каждой модели самолета;
- сколько посадочных талонов выпущено на один рейс.

Каждый запрос должен иметь понятные имена столбцов и детерминированный `ORDER BY`, если используется `LIMIT`.

### 5. Проверки качества данных

В `q04_quality_checks.sql` напишите минимум 3 SQL-проверки.

Обязательная проверка для всех вариантов:

- найти бронирования, где `bookings.total_amount` не совпадает с суммой `segments.price` по билетам этого бронирования.

Дополнительно выберите две проверки, релевантные вашему варианту:

- рейсы с некорректной парой `actual_departure` / `actual_arrival`;
- билеты без сегментов;
- сегменты без посадочного талона для уже завершенных рейсов;
- посадочные талоны, где место не входит в конфигурацию самолета;
- аэропорты с одинаковым городом и страной;
- маршруты, где аэропорт отправления совпадает с аэропортом прибытия;
- любые другие проверки, если вы объясните их смысл.

Проверка может вернуть ноль строк. Это нормальный результат, если вы объяснили, какое правило проверялось.

## answers.md

Ответьте:

1. Какой фрагмент предметной области описывают таблицы вашего варианта?
2. Какие связи являются обязательными, а какие допускают отсутствие связанных строк?
3. Какие бизнес-правила выражены ограничениями БД?
4. Какие правила не проверяются схемой напрямую?
5. Какая из ваших SQL-проверок кажется самой важной и почему?

## Ограничения

- Не используйте `SELECT *` в итоговых запросах.
- Не пишите проверки только ради количества: каждая проверка должна иметь предметный смысл.
- Не утверждайте, что правило гарантируется БД, если вы видите его только по текущим данным.
