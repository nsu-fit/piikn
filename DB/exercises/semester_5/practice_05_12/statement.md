# Практика 5.12. Представления и простые функции

Вы оформляете повторяемые отчеты и одну проверку качества данных как объекты БД. Цель - не перенести всю бизнес-логику в БД, а показать аккуратное повторное использование SQL.

## Что нужно сдать

```text
semester-5/practice-12-views-functions/
  README.md
  views.sql
  functions.sql
  usage.sql
```

## Подготовка

Работайте в схеме `course_lab`. Исходные данные берите из учебной схемы `bookings`.

Каждый файл должен быть воспроизводимым: используйте `CREATE OR REPLACE VIEW`, `CREATE OR REPLACE FUNCTION` и явное указание схем.

## `views.sql`

Создайте два представления.

### `course_lab.v_route_revenue`

Столбцы:

- `route_no`;
- `departure_airport`;
- `arrival_airport`;
- `segment_count`;
- `total_revenue`.

Смысл: суммарная выручка и количество сегментов по маршруту.

### `course_lab.v_booking_quality`

Столбцы:

- `booking_ref`;
- `total_amount`;
- `calculated_amount`;
- `amount_delta`.

Смысл: сравнение суммы бронирования с суммой сегментов входящих билетов.

## `functions.sql`

Создайте функцию:

```sql
course_lab.booking_total_delta(p_booking_ref text) returns numeric
```

Функция должна возвращать `total_amount - sum(segment.price)` для заданного бронирования.

Требования:

- если бронирование не найдено, функция возвращает `NULL`;
- если у бронирования нет сегментов, сумма сегментов считается `0`;
- функция не изменяет данные.

## `usage.sql`

Добавьте демонстрационные запросы:

- `q01_top_routes`: 10 маршрутов с максимальной `total_revenue`;
- `q02_booking_mismatches`: бронирования, где `amount_delta <> 0`;
- `q03_function_existing_booking`: вызов функции для существующего `booking_ref`;
- `q04_function_missing_booking`: вызов функции для несуществующего `booking_ref`;
- `q05_view_reuse`: запрос, который использует `v_route_revenue` как обычную таблицу.
