# Ожидаемые свойства решения

В корректном решении:

- `changes.sql` можно выполнить после `provided/seed_dml_lab.sql`;
- кандидаты на обработку сначала показаны через `SELECT`;
- audit-вставка не создает дубликаты при повторном запуске;
- `UPDATE ... FROM` меняет только ожидаемые `pending`-заказы;
- `RETURNING` используется для фиксации измененных строк;
- старые `draft` переводятся в `cancelled`, а не удаляются физически;
- `checks.sql` показывает инварианты после сценария.

Не требуется:

- писать DDL для `order_queue` и `order_audit`;
- использовать CTE или оконные функции;
- работать с таблицами демобазы напрямую.
