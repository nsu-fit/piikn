# Лекция 6.5. Оптимизатор запросов

Место в курсе: лекция объясняет, как СУБД выбирает план выполнения и как инженер должен читать этот план.

## Цели

- Разобрать назначение cost-based optimizer.
- Объяснить статистику, оценку кардинальности и стоимость операций.
- Показать основные алгоритмы соединения.
- Подготовить студентов к чтению `EXPLAIN ANALYZE` для операторских и прикладных сценариев.

## План лекции

1. Почему декларативный SQL требует оптимизатора.
2. План выполнения: дерево операторов, потоки строк, стоимость.
3. Статистика таблиц: количество строк, распределение значений, most common values, гистограммы.
4. Cardinality estimation: почему ошибка в оценке строк ломает весь план.
5. Access methods: sequential scan, index scan, bitmap scan.
6. Join algorithms: nested loop, hash join, merge join.
7. Сортировка, агрегирование, hash aggregate, group aggregate.
8. `EXPLAIN` и `EXPLAIN ANALYZE`: оценка и фактическое выполнение.
9. Что искать в плане: расхождение estimated/actual rows, лишние loop, large sort, disk spill, sequential scan по большой таблице.
10. Как улучшать запрос: индекс, переписывание, статистика, ограничения, изменение гранулярности.

## Ключевые понятия

Optimizer, execution plan, cost, cardinality estimation, statistics, sequential scan, index scan, nested loop, hash join, merge join, `EXPLAIN ANALYZE`.

## Связь с практикой

Студенты читают `EXPLAIN ANALYZE` для операторских и прикладных сценариев: поиск задач выполнения, состояний окружений, переиспользуемых состояний и audit-записей.

## Контрольные вопросы: что вы должны были узнать за эту лекцию

1. Почему СУБД сама выбирает план выполнения запроса?
2. Что показывает `EXPLAIN`, а что добавляет `EXPLAIN ANALYZE`?
3. Почему оценка кардинальности критична для выбора плана?
4. Чем nested loop отличается от hash join?
5. Какие признаки в плане указывают на проблему производительности?
6. Какие способы улучшения запроса нужно рассматривать до добавления нового индекса?
