# Архитектура практических упражнений и контрольных заданий

Документ задает общий подход к разработке конкретных упражнений для практических занятий, контрольных работ и проектных checkpoint-артефактов по дисциплине "Базы данных".

Он дополняет:

- `DB/practical_classes_plan.md`;
- `DB/practical_assessment_methodology.md`;
- `DB/practical_plans/`;
- `DB/updated_database_course_program.md`.

## 1. Цели архитектуры упражнений

Практические задания должны проверять не только факт запуска SQL, но и переносимость навыка:

- студент должен решать задачу по схеме и требованиям, а не подгонять ответ под конкретный маленький набор строк;
- одно и то же решение должно работать на базе другого размера и с другими данными той же схемы;
- автоматическая проверка должна отсекать синтаксические ошибки, хардкод и невоспроизводимые решения;
- ручная проверка должна оставаться там, где требуется инженерное объяснение: модель предметной области, нормализация, транзакции, безопасность, ADR, runbook.

## 2. Основной набор данных

Основной базой для SQL-практики 5-го семестра и части PostgreSQL-практик 6-го семестра предлагается сделать демонстрационную базу Postgres Pro "Авиаперевозки".

Причины выбора:

- база официально предназначена для обучения SQL и подготовки учебных материалов;
- распространяется под MIT license;
- имеет несколько размеров с одинаковой схемой;
- поддерживается для PostgreSQL 15+;
- содержит достаточно реалистичную предметную область;
- включает не только простые таблицы, но и PostgreSQL-специфичные элементы: JSONB, диапазоны, массивы, views, constraints, indexes.

Текущая версия демобазы Postgres Pro от 01.09.2025 доступна в четырех размерах:

| Вариант | Сжатый SQL dump | Примерный размер БД | Основное применение |
|---|---:|---:|---|
| 3 months | 133 MB | 1.3 GB | Обычные практики, первые SQL-запросы, локальный запуск |
| 6 months | 276 MB | 2.7 GB | Домашние проверки, более заметные планы запросов |
| 1 year | 558 MB | 5.4 GB | Скрытые проверки и оптимизационные задания |
| 2 years | 1137 MB | 11 GB | Демонстрации больших объемов, продвинутые проверки, shared/Taidon-контур |

Базовая схема включает таблицы и представления:

- `bookings`, `tickets`, `segments`;
- `flights`, `routes`, `timetable`;
- `airports_data`, `airports`;
- `airplanes_data`, `airplanes`;
- `seats`, `boarding_passes`.

Сильные стороны для курса:

- естественные `JOIN`-цепочки: бронирование - билеты - сегменты - рейсы - маршруты - аэропорты;
- агрегаты по времени, маршрутам, аэропортам, классам обслуживания и статусам рейсов;
- временная логика: расписание, фактические вылеты/прилеты, задержки, validity windows;
- представления, скрывающие часть сложности;
- JSONB-поля для многоязычных названий;
- массивы и range-типы;
- ограничения и индексы, пригодные для анализа;
- разные размеры базы для проверки переносимости решения.

Ограничения:

- предметная область одна, поэтому есть риск "натаскать" студентов на одну схему;
- мало естественных иерархий для рекурсивных CTE;
- нет полноценного приложения и явного слоя миграций;
- не все темы нормализации удобно показывать на уже аккуратно спроектированной схеме;
- транзакционные конфликты, SQL injection и миграции лучше давать на специально подготовленных учебных таблицах.

Вывод: Postgres Pro demo database должна быть основной, но не единственной базой упражнений.

## 3. Расширяющий учебный слой

Чтобы закрыть темы, которые плохо ложатся на авиаперевозки, поверх основной БД нужен небольшой `course_lab` schema extension pack.

Он должен быть маленьким, воспроизводимым и генерируемым:

- `course_lab.org_units` и `course_lab.employees` - иерархии и рекурсивные CTE;
- `course_lab.product_categories`, `course_lab.products`, `course_lab.orders`, `course_lab.order_items` - нормализация, DDL/DML, транзакции, инварианты;
- `course_lab.account_ledger` - транзакционные сценарии, lost update, write skew, ограничения баланса;
- `course_lab.event_log` - partitioning, retention, BRIN, аналитические витрины;
- `course_lab.documents` - полнотекстовый поиск, JSONB metadata, опциональный vector prototype;
- `course_lab.denormalized_ticket_manifest` - специально денормализованный extract из flights-базы для функциональных зависимостей и нормализации.

Расширяющий слой не должен заменять основную демобазу. Его назначение - давать компактные управляемые сценарии, где нужно:

- создать схему с нуля;
- намеренно нарушить нормальные формы;
- воспроизвести конфликт транзакций;
- проверить миграцию;
- построить partitioning/retention;
- тестировать безопасность динамического SQL.

## 4. Политика размеров данных

### 5-й семестр

Обычные практики:

- видимая база: Postgres Pro demo 3 months;
- в Taidon можно использовать тот же dump или подготовленный snapshot;
- упражнения должны требовать SQL-решение, а не ручной подбор результата.

Домашняя автопроверка:

- запуск на 3 months;
- периодическая дополнительная проверка на 6 months;
- для сложных запросов допустим таймаут, чтобы студент видел цену неэффективного решения.

Контрольные:

- студент работает с видимой учебной базой и индивидуальным вариантом;
- проверка решения может выполняться на базе большего размера или на скрытой базе той же схемы;
- задания формулируются так, чтобы правильный SQL переносился между размерами.

### 6-й семестр

Локальные практики:

- обязательный контур: 3 months или специально уменьшенный локальный snapshot;
- для работ с файлами кластера, backup/restore, ролями и миграциями размер должен быть посильным для ноутбука.

Оптимизация и планы:

- базовый вариант: 3 months;
- продвинутый или скрытый вариант: 6 months / 1 year;
- 2 years используется только для shared/Taidon-контура, демонстраций преподавателя или добровольных продвинутых задач.

## 5. Варианты заданий

Обычные практики:

- постановка общая для всей группы;
- параметры разные: аэропорт, страна, период, класс обслуживания, статус рейса, route number, threshold, набор таблиц;
- параметры выдаются из `variant_seed`, например по номеру студента или случайному seed.

Контрольные:

- задания строятся из пула шаблонов;
- каждый студент получает индивидуальные параметры;
- для ключевых контрольных желательно иметь 2-3 предметных фрагмента, чтобы не было одного заученного пути решения.

Проекты:

- тема выбирается из каталога допустимых проектных типов;
- требования одинаковы по структуре: проблема, артефакт, проверка, риски, защита;
- конкретный dataset, запрос, миграция или эксплуатационная процедура различаются.

## 6. Защита от подгонки ответов

Нельзя принимать решения, завязанные на конкретный маленький результат:

- hard-coded списки id, аэропортов, ticket numbers и сумм без обоснования запрещены;
- `LIMIT` без детерминированного `ORDER BY` считается ошибкой, если результат должен быть воспроизводимым;
- решение должно работать на другом размере базы;
- для проверок используются hidden tests: большая база, дополнительные строки или другой seed extension pack;
- для агрегатов и отчетов проверяется не только количество строк, но и ключевые значения;
- для DDL/DML проверяются ограничения и инварианты, а не только факт создания таблицы.

Типовая схема:

1. Студент видит маленькую базу и формулировку.
2. Студент сдает SQL/Markdown.
3. Автопроверка запускает решение на видимой базе.
4. Для контрольных и части портфолио решение дополнительно запускается на скрытой базе той же схемы.
5. На защите преподаватель спрашивает, почему запрос переносим между размерами.

## 7. Автоматическая проверка

Автоматика должна проверять то, что хорошо формализуется:

- SQL-файл выполняется без ошибок;
- результат запроса совпадает с эталоном;
- результат сравнивается как множество строк, если порядок не является частью задания;
- порядок проверяется только при явном `ORDER BY`;
- DDL создает ожидаемые таблицы, типы, ключи, внешние ключи, `CHECK` и `UNIQUE`;
- DML приводит к ожидаемому состоянию;
- миграция проходит на исходных тестовых данных;
- rollback/forward-fix запускается;
- backup восстанавливается и проходит smoke checks;
- `EXPLAIN ANALYZE` приложен и относится к нужному запросу;
- структура папки и README соответствуют шаблону.

Автопроверка не должна пытаться полностью оценить:

- качество ER-диаграммы;
- корректность всех функциональных зависимостей;
- инженерную убедительность нормализации;
- объяснение транзакционной аномалии;
- безопасность архитектурного решения;
- качество ADR/runbook;
- устную защиту.

Эти части проверяются вручную или через короткое устное ревью.

## 8. Технический формат проверки

Рекомендуемый формат задания:

```text
exercise/
  statement.md
  variant.yaml
  seed/
  expected/
  tests/
  README.md
```

Рекомендуемый формат сдачи:

```text
semester-5/practice-07-dml-select/
  solution.sql
  answers.md
  README.md
```

Для SQL-заданий желательно задавать имена файлов и запросов:

- `q01.sql`, `q02.sql`, `q03.sql` - отдельные запросы;
- `solution.sql` - если порядок выполнения важен;
- `answers.md` - объяснения и ответы на открытые вопросы;
- `checks.sql` - самопроверка студента;
- `report.md` - выводы по планам, индексам, backup/restore или миграции.

Автопроверка должна запускаться в чистой транзакции или в пересоздаваемом snapshot, чтобы работы студентов не влияли друг на друга.

## 9. Типы упражнений по семестрам

### 5-й семестр

Основная линия: Postgres Pro demo database в Taidon/sqlrs.

Первые три недели не проверяют самостоятельное написание SQL. Студенты запускают готовые обзорные и диагностические запросы, читают результаты, строят словарь данных и ER-фрагменты. Практика 5.5 также остается без самостоятельного SQL: студенты работают с реляционной алгеброй на уровне выражений и табличных результатов. Это нужно, чтобы практика соответствовала порядку лекций: прикладной `SELECT`, DML и минимальный DDL появляются в заданиях только после соответствующих лекций.

| Блок | Типовые упражнения | Данные |
|---|---|---|
| Чтение схемы | найти сущности, ключи, связи, неявные правила | flights 3 months |
| ER-реконструкция | восстановить фрагмент бронирование - билет - сегмент - рейс | flights 3 months |
| Реляционная алгебра | построить выражение, вручную получить результат на малых отношениях, сравнить эквивалентные формы | малые табличные фрагменты flights/course_lab |
| Базовый SQL | отчеты по маршрутам, аэропортам, статусам, классам обслуживания | flights 3 months, hidden 6 months |
| DML на основе SELECT | перенос данных, корректирующие обновления, мягкое удаление, `RETURNING`, инварианты | `course_lab` |
| DDL | учебный фрагмент заказов или таблица расширения, готовые проверки ограничений | `course_lab` |
| Нормализация | денормализованный manifest по билетам и рейсам | `course_lab.denormalized_ticket_manifest` |
| Транзакции | бронирование места, изменение баланса, concurrent check-in | `course_lab.account_ledger` или `course_lab.seat_reservation_lab` |
| Безопасный SQL | исправление unsafe filter/order/search snippets | `course_lab` + псевдокод |

### 6-й семестр

Основная линия: локальный/containerized PostgreSQL.

| Блок | Типовые упражнения | Данные |
|---|---|---|
| Развертывание | поднять PostgreSQL, роли, seed, smoke checks | flights 3 months |
| Storage internals | relation files, `pg_class`, размеры, TOAST | flights + `course_lab` |
| Индексы | B-tree, partial/expression, GIN, BRIN, full-text | flights 3/6 months |
| Оптимизация | before/after, `EXPLAIN ANALYZE`, индексы, переписывание | flights 6 months / hidden 1 year |
| WAL/recovery | backup/restore, smoke checks, recovery notes | локальный контур |
| Репликация | primary/replica, lag, read-only запросы | контейнерный контур |
| Администрирование | роли, config, health checks, slow query diagnostics | локальный контур |
| Advanced SQL в приложении | keyset pagination, таблица как очередь, REST-idempotency | `course_lab` + мини-приложение |
| Миграции | expand-contract, locks, backfill, forward-fix | `course_lab` |
| Partitioning | event_log, retention, BRIN | `course_lab.event_log` |
| JSONB/модели и access policies | normalized vs JSONB vs object reference, RBAC/RLS sketch | `course_lab.documents` |
| Граф | recursive CTE, adjacency list, materialized path | `course_lab.org_units` |
| Аналитика | star schema, window functions, materialized views, refresh | flights + `course_lab.event_log` |
| Поиск/AI | full-text, optional pgvector prototype | `course_lab.documents` |

## 10. Альтернативные базы

Postgres Pro demo database рекомендуется как основная. Альтернативы полезны как дополнительные источники идей, но не должны становиться второй обязательной линией курса без отдельного решения.

| База | Плюсы | Минусы | Рекомендация |
|---|---|---|---|
| Pagila/Sakila | Хорошо знакомая rental/e-commerce структура, много joins | Обычно небольшая, менее естественна для больших планов и storage experiments | Использовать как источник отдельных SQL-задач, не как основу |
| Chinook | Очень проста, удобна для первых запросов | Слишком маленькая и узкая для курса на два семестра | Не нужна, если есть flights |
| Northwind | Классический orders/products пример | Устаревшая модель, мало PostgreSQL-специфики | Только для сравнения или исторического примера |
| AdventureWorks/PostgreSQL ports | Богатая enterprise-модель | Сложнее вводить, порты неоднородны, чаще SQL Server-centered | Не брать как базовую без сильной причины |
| TPC-H/TPC-DS | Стандартизованы для аналитики и оптимизации | Искусственная предметная область, плохо подходят для моделирования и нормализации | Использовать точечно в 6-м семестре для OLAP/optimizer |
| StackOverflow/IMDb/NYC Taxi | Реальные и большие данные | Сложность импорта, лицензии, приватность, несогласованная схема | Только как факультативные продвинутые datasets |

## 11. Решение для текущего курса

Базовое решение:

1. Взять Postgres Pro demo database "Авиаперевозки" как основной dataset.
2. Для практик 5-го семестра использовать 3 months в Taidon/sqlrs.
3. Для скрытых проверок использовать 6 months или 1 year.
4. Для 6-го семестра использовать 3 months локально/containerized; 6 months или 1 year для оптимизации и скрытых проверок.
5. Добавить `course_lab` extension pack для нормализации, рекурсивных CTE, транзакций, миграций, partitioning, search и эксплуатационных сценариев.
6. Все задания проектировать параметризуемыми, с variant seed и hidden tests.
7. Автоматикой проверять SQL, DDL/DML, структуру сдачи и smoke checks; вручную проверять объяснения, ER/нормализацию, ADR/runbook и защиту.
