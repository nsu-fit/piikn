# Решение 5.2. Анализ учебного датасета и словарь данных

## Эталонная интерпретация

Для любого варианта ответ должен разделять четыре слоя:

- `columns`: типы, nullable, смысл ключевых атрибутов;
- `constraints`: первичные, внешние, уникальные и проверочные ограничения;
- `cardinality`: распределения вроде "несколько билетов в бронировании" или "несколько сегментов в билете";
- `quality checks`: что проверял диагностический запрос и почему ноль строк тоже является результатом.

## Типовые связи

- `bookings` 1:N `tickets`: одно бронирование может содержать несколько билетов.
- `tickets` 1:N `segments`: билет может включать несколько flight segments.
- `flights` 1:N `segments`: один рейс продается многим билетам.
- `flights` 1:N `boarding_passes`: на рейс выдается много посадочных талонов.
- `routes` 1:N `flights` с учетом temporal-validity: рейс использует маршрут, действующий на момент вылета.
- `airplanes_data` 1:N `seats`: конфигурация мест принадлежит типу самолета.

## Правильные формулировки

- "Нарушений не найдено на текущих данных" корректно.
- "Схема гарантирует невозможность нарушения" корректно только при наличии constraint.
- "Похоже, правило поддерживается приложением или процессом" корректно для правил, которые видны по данным, но не выражены ограничением.

## Частые ошибки

- Перевод имени столбца вместо предметного смысла.
- Смешение `airports` view и `airports_data` table.
- Игнорирование `validity` у маршрутов.
- Вывод бизнес-правила только из одного наблюдаемого примера.
